Part A – Theory
1. Define Constraint.

Answer:
A constraint is a rule added to a table to control what data can be stored. It helps keep the data correct and valid.

2. Define Primary Key.

Answer:
A Primary Key is a column (or columns) that uniquely identifies each record in a table. It cannot have duplicate or NULL values.

3. Define Foreign Key.

Answer:
A Foreign Key is a column that connects one table with another. It makes sure the value exists in the parent table.

4. Define Candidate Key.

Answer:
A Candidate Key is any column that can uniquely identify a record. One of the candidate keys is chosen as the Primary Key.

5. Define Alternate Key.

Answer:
An Alternate Key is a candidate key that is not selected as the Primary Key. It still contains unique values.

6. Define Composite Key.

Answer:
A Composite Key is made by combining two or more columns to uniquely identify each record.

7. Define Super Key.

Answer:
A Super Key is any column or combination of columns that can uniquely identify a row. It may contain extra columns.

8. Define Unique Key.

Answer:
A Unique Key makes sure all values in a column are different. It does not allow duplicate values.

9. Define NOT NULL Constraint.

Answer:
The NOT NULL constraint makes sure a column cannot have empty (NULL) values.

10. Define CHECK Constraint.

Answer:
A CHECK constraint allows only values that satisfy a given condition.

11. Define DEFAULT Constraint.

Answer:
A DEFAULT constraint automatically inserts a value if no value is given during insertion.

12. Why are constraints required?

Answer:
Constraints are required to keep the data accurate, valid, and consistent. They also help prevent incorrect or duplicate data.


Part B – Compare
1. Primary Key vs Unique Key

A Primary Key is used to identify each record in a table, and it cannot contain duplicate or NULL values. A Unique Key also prevents duplicate values, but it can allow NULL values depending on the database.

2. Primary Key vs Foreign Key

A Primary Key uniquely identifies each record in its own table. A Foreign Key is used to connect one table with another and refers to the Primary Key of another table.

3. Candidate Key vs Super Key

A Candidate Key is the smallest set of columns that can uniquely identify a record. A Super Key can also identify a record, but it may contain extra columns that are not needed.

4. Candidate Key vs Alternate Key

A Candidate Key is any column that can uniquely identify a record. An Alternate Key is a Candidate Key that was not selected as the Primary Key.

5. Composite Key vs Primary Key

A Composite Key is made by combining two or more columns to identify a record. A Primary Key is the main key of the table and can be either a single column or a combination of columns.

6. Constraint vs Key

A Constraint is a rule that controls what data can be stored in a table. A Key is used to identify records or create relationships between tables.

7. NOT NULL vs CHECK

The NOT NULL constraint makes sure a column cannot have empty values. The CHECK constraint makes sure the value follows a specific condition.

8. UNIQUE vs PRIMARY KEY

The UNIQUE constraint makes sure that duplicate values are not allowed in a column. The PRIMARY KEY also prevents duplicate values, but it does not allow NULL values and is used to identify each record in the table.

Part D- Practical

1. Create Database
CREATE DATABASE company_db;
USE company_db;

2. Create Department Table
CREATE TABLE Department (
    Department_ID INT PRIMARY KEY,
    Department_Name VARCHAR(50) NOT NULL UNIQUE,
    Location VARCHAR(50) NOT NULL
);

3. Create Employee Table (with all constraints)
CREATE TABLE Employee (
    Employee_ID INT PRIMARY KEY,
    Employee_Name VARCHAR(100) NOT NULL,
    Email VARCHAR(100) NOT NULL UNIQUE,
    PAN_Number VARCHAR(10) NOT NULL UNIQUE,
    Gender VARCHAR(10) NOT NULL CHECK (Gender IN ('Male','Female','Other')),
    Salary DECIMAL(10,2) NOT NULL CHECK (Salary > 0),
    City VARCHAR(50) DEFAULT 'Chennai',
    Joining_Date DATE NOT NULL,
    Department_ID INT NOT NULL,
    FOREIGN KEY (Department_ID) REFERENCES Department(Department_ID)
);

4. Create Project Table
CREATE TABLE Project (
    Project_ID INT PRIMARY KEY,
    Project_Name VARCHAR(100) NOT NULL UNIQUE,
    Budget DECIMAL(12,2) NOT NULL CHECK (Budget > 0)
);

5. Create Employee_Project Table using Composite Key
CREATE TABLE Employee_Project (
    Employee_ID INT,
    Project_ID INT,
    Assigned_Date DATE DEFAULT (CURRENT_DATE),
    PRIMARY KEY (Employee_ID, Project_ID),
    FOREIGN KEY (Employee_ID) REFERENCES Employee(Employee_ID),
    FOREIGN KEY (Project_ID) REFERENCES Project(Project_ID)
);

Part E – Constraint Testing
1. Insert duplicate Primary Key. Did it work? Why?

Answer:
No, it did not work because a Primary Key must be unique. Duplicate values are not allowed.

2. Insert duplicate UNIQUE value.

Answer:
It did not work because the UNIQUE constraint does not allow duplicate values.

3. Insert NULL into NOT NULL column.

Answer:
It did not work because a NOT NULL column cannot store an empty value.

4. Insert invalid FOREIGN KEY.

Answer:
It did not work because the Foreign Key value must exist in the parent table.

5. Insert invalid CHECK value.

Answer:
It did not work because the value did not satisfy the CHECK condition.

6. Insert record without DEFAULT column.

Answer:
It worked because MySQL automatically inserted the default value.

7. Record MySQL error messages.

Answer:
MySQL showed error messages saying the constraint was violated, such as duplicate entry, foreign key constraint failed, or cannot be NULL.

Part F – Interview
1. Can a table have multiple Primary Keys?

Answer:
No. A table can have only one Primary Key.

2. Can a table have multiple Candidate Keys?

Answer:
Yes. A table can have more than one Candidate Key.

3. Can a Foreign Key contain duplicate values?

Answer:
Yes. A Foreign Key can have duplicate values.

4. Can UNIQUE contain NULL?

Answer:
Yes. A UNIQUE column can contain NULL values.

5. Can Composite Key have more than two columns?

Answer:
Yes. A Composite Key can be made of two or more columns.

6. Is every Candidate Key a Super Key?

Answer:
Yes. Every Candidate Key is also a Super Key.

7. Is every Super Key a Candidate Key?

Answer:
No. A Super Key may contain extra columns, so it is not always a Candidate Key.

8. What is Referential Integrity?

Answer:
Referential Integrity makes sure the relationship between tables stays correct.

9. Which constraint maintains Referential Integrity?

Answer:
The Foreign Key constraint maintains Referential Integrity.
